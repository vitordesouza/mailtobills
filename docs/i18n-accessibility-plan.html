Assessment

i18n and accessibility plan

MailToBills should add internationalization and accessibility as one coordinated initiative, but not as one large product-wide pull request. Keep the work small, route-safe, and tied to the existing landing, dashboard, and feature-module boundaries.

PR recommendation

Do not ship the full i18n and accessibility migration in a single PR. Ship a foundation PR, then pair i18n and accessibility changes per surface so copy, accessible names, and tests move together.

Best i18n path

Use next-intl in both Next apps. It matches the App Router, Server Components, ICU messages, typed keys, and formatting needs without changing the product stack.

Next App Router ICU messages Typed keys

Best a11y path

Add static linting with eslint-plugin-jsx-a11y, runtime scans with @axe-core/playwright, and focused manual checks for keyboard, focus, screen reader flow, and PDF/download actions.

WCAG A/AA gate Keyboard first

First locales

Keep en as default and add pt-PT first. This fits the current product language, existing copy mentions of fatura, and the Lisbon/Portugal operating context.

Default: en First add: pt-PT

Technology Stack

  • i18n runtime: next-intl, installed in apps/landing and apps/dashboard.
  • Shared locale model: add a small packages/i18n only for locale constants, labels, validation, and formatting defaults. Do not put app copy there.
  • Accessibility lint: add eslint-plugin-jsx-a11y to packages/eslint-config/next.js, using flat config and component settings for local UI primitives where useful.
  • Accessibility runtime tests: add @axe-core/playwright to root E2E tests and scan landing, dashboard sign-in, and later authenticated dashboard routes.

Routing Decision

  • Landing: use locale routes because it is public, SEO-facing, and content-led. Keep default English available at /; add Portuguese at /pt or /pt-PT after choosing URL style.
  • Dashboard: avoid locale-prefixed routes at first. Keep /signin, /settings, and /m/[month] stable because auth middleware and sidebar navigation currently assume unprefixed paths.
  • Dashboard locale source: read user preference from Convex when authenticated, with a cookie fallback for /signin and first visit.

Organisation

Concern Recommended Location Reason
Locale constants packages/i18n/src/locales.ts Shared by landing, dashboard, tests, and future outbound copy without coupling UI messages to domain types.
Landing messages apps/landing/messages/{locale}/... Landing copy is marketing copy and should stay with the landing app, split by page section if files grow.
Dashboard shell messages apps/dashboard/messages/{locale}/common.json Navigation, settings shell, auth copy, and shared labels can be loaded once through the dashboard provider.
Feature messages apps/dashboard/features/expense-documents/messages/{locale}.json Keeps Expense Document wording and accessible labels beside the feature module that owns the rows, actions, and read model.
Accessible component defaults packages/ui/src/components Keep focus styles, labels, descriptions, and semantic behavior close to the reusable primitives.
E2E a11y scans e2e/accessibility.spec.ts Root Playwright already runs against landing. Extend it first, then add dashboard routes when authenticated fixtures are available.

i18n Implementation Shape

  • Add Locale, defaultLocale, supportedLocales, and isLocale in packages/i18n.
  • Configure next-intl request loading separately in each app so landing and dashboard can load different message bundles.
  • Move visible strings, metadata, button labels, form labels, status text, empty states, and aria-label values into translation namespaces.
  • Use locale-aware formatting for collection months, timestamps, counts, and prices. Do not localize stored IDs, file names, route params, Convex function names, or CSV field contracts in the first pass.
  • Replace the disabled language select in dashboard preferences with a real setting only after the provider and fallback behavior are in place.

Accessibility Implementation Shape

  • Add a skip link and confirm one meaningful main landmark per page shell.
  • Audit icon-only and icon-leading controls for stable accessible names, especially month navigation, table expansion, delete, download, export, sidebar toggle, and language selection.
  • Verify forms expose labels, descriptions, invalid states, and error messages through semantic relationships.
  • Keep focus visible and predictable through menus, dialogs, dropdowns, table row expansion, and mobile sidebar states.
  • Add Playwright axe scans for public pages and smoke-level manual keyboard scripts for dashboard workflows that need auth.

Pull Request Plan

PR 1

Foundation: no broad copy migration

Add packages/i18n, install/configure next-intl, add eslint-plugin-jsx-a11y, add @axe-core/playwright, and create the first accessibility test fixture.

  • Keep routes and visible copy mostly unchanged.
  • Verify pnpm lint, pnpm typecheck, and E2E smoke.
  • Expected review size: configuration and guardrails.
PR 2

Landing: public i18n plus public a11y

Localize the landing app, metadata, header/footer, sections, and CTAs. Add locale-aware public routing and axe scans for each public locale.

  • Keep / stable for default English.
  • Translate accessible names at the same time as visible copy.
  • Verify SEO metadata, language switcher, keyboard navigation, and contrast.
PR 3

Dashboard shell: provider, settings, auth, navigation

Add dashboard locale provider, persist the selected locale, localize sign-in, sidebar, settings, billing, forwarding addresses, and preference copy.

  • Keep dashboard URLs unprefixed.
  • Make the language select functional with cookie fallback.
  • Run axe on /signin and manual keyboard checks on authenticated shell paths.
PR 4

Expense Documents feature

Localize the month dashboard, reports page, table actions, attachment labels, status text, empty states, and export/send controls inside the existing features/expense-documents module.

  • Use locale-aware month and timestamp formatting.
  • Do not introduce new document states while translating old copy.
  • Extend component tests for accessible names and expanded row behavior.
PR 5

Outbound copy and export policy

Decide whether transactional email copy and Accountant Export filenames are locale-specific. Keep Manifest column names stable unless there is a deliberate accountant-facing compatibility decision.

  • Localize customer-facing email subject/body copy.
  • Keep machine-readable export contracts stable by default.
  • Document any locale-specific export behavior in the operations runbook.

Primary Risks

  • Locale-prefixed dashboard routes would require auth middleware, sidebar, redirects, tests, and deep links to change together.
  • Large copy extraction can hide behavior changes in review. Keep visible copy migration separate from route and auth changes.
  • Automated accessibility scans are useful but incomplete. They must be paired with keyboard and screen reader spot checks.
  • Domain terms must stay consistent: use Expense Document, Customer, Collection Month, Primary Attachment, and Accountant Export.

Definition of Done

  • html lang, metadata, visible copy, and accessible labels all reflect the selected locale.
  • pnpm lint, pnpm typecheck, app builds, Vitest, and Playwright pass.
  • Axe scans pass for landing locales and dashboard sign-in.
  • Manual keyboard path passes for sign-in, sidebar navigation, month navigation, table expansion, export/download, settings, and language selection.

Should Both Ship in a Single PR?

No, not as a full implementation. i18n touches nearly every visible string, metadata value, route decision, and accessible label. Accessibility touches semantics, focus behavior, forms, tests, and component primitives. A single product-wide PR would be difficult to review and risky to revert.

The practical split is: one small foundation PR for shared tooling, then surface-by-surface PRs where i18n and accessibility move together. That keeps accessible names translated with their visible labels without mixing landing SEO, dashboard auth, and Expense Document workflow changes in one review.
